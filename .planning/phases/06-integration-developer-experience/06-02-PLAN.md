<task type="auto">
  <name>Task 1: Create error types and formatters</name>
  <files>src/carl/errors.ts</files>
  <action>
Create structured error handling module:

1. Define error categories enum:
   ```typescript
   export type CarlErrorCategory = 'config' | 'manifest' | 'runtime' | 'permission';
   ```

2. Define `CarlError` class extending Error:
   - category: CarlErrorCategory
   - context: string (what was happening)
   - location: string (file or operation that failed)
   - fix: string (copy-pasteable suggestion)
   - docsLink?: string (link to troubleshooting guide section)

3. Implement `formatError(error: unknown): string`:
   - Handle CarlError instances with full formatting
   - Handle generic errors with fallback formatting
   - Output format: `[carl] ERROR: {context}\n  Location: {location}\n  Problem: {message}\n  Fix: {fix}\n  Docs: {docsLink}`
   - Use console.error-friendly format

4. Pre-defined error factory functions:
   - `manifestParseError(path: string, details: string): CarlError`
   - `domainLoadError(domain: string, path: string, details: string): CarlError`
   - `permissionError(path: string, operation: string): CarlError`
   - `configReadError(path: string, details: string): CarlError`

All fix suggestions should be actionable commands like `Check file permissions: ls -la {path}` or `Validate manifest: cat {path}`.
  </action>
  <verify>
    <automated>node -e "const e = require('./dist/carl/errors.js'); const err = new e.CarlError('test', 'config', 'testing', 'here', 'fix it'); console.log(e.formatError(err))"</automated>
  </verify>
  <done>Error module exports CarlError, formatError, and error factory functions</done>
</task>

<task type="auto">
  <name>Task 2: Integrate errors into loader and hooks</name>
  <files>src/carl/loader.ts, src/integration/plugin-hooks.ts</files>
  <action>
Integrate structured errors into existing error handling:

1. In loader.ts:
   - Import CarlError and factory functions
   - Replace generic warnings in loadProjectRules with CarlError throws where critical failures occur
   - Keep non-fatal warnings as warnings (don't throw for parse warnings)
   - Wrap file read operations with try/catch using permissionError/configReadError

2. In plugin-hooks.ts:
   - Import formatError from errors module
   - Wrap the experimental.chat.system.transform hook body in try/catch
   - On caught error, console.error(formatError(error))
   - Continue gracefully where possible (partial functionality better than crash)

3. Add error handling to command handlers:
   - Wrap /carl setup command in try/catch
   - Wrap /carl setup --integrate-opencode in try/catch
   - Display formatted errors to user

Pattern: Errors should fail the specific operation but not crash the entire plugin.
